# Оптимізація `ПолучитьТоварыПакетом`

Готова функція для вставлення в модуль 1С:
[`ПолучитьТоварыПакетом_оптимизированная.bsl`](./ПолучитьТоварыПакетом_оптимизированная.bsl).

Цей документ описує сумісний контракт для прискорення `allgoods` без передачі
Base64 у списку товарів. Поточний Next.js уже надсилає `ВключатьФотоBase64 =
Ложь`, тому зображення мають завантажуватися окремим пакетним endpoint.

## Критичні зміни

1. Звичайна сторінка спочатку відбирає лише `Лимит + 1` посилань
   номенклатури, а ціни й залишки агрегуються тільки для цієї тимчасової
   таблиці.
2. Для фільтрації або сортування за ціною/залишком спочатку формується повна
   тимчасова таблиця кандидатів за дешевими фасетами, а вже потім до неї
   приєднуються ціни й залишки.
3. `COUNT` виконується лише на першій сторінці або коли явно передано
   `ВключатьОбщееКоличество = Истина`.
4. Фасети `Категория`, `Группа`, `Подгруппа` і виробник порівнюються через `=`,
   а не `ПОДОБНО Значение + "%"`.
5. `Описание` фільтрується на сервері, але повертається тільки при
   `ВключатьОписание = Истина`.
6. `ТолькоСЦеной` перевіряє публічну `ЦінаПрод > 0`. Закупівельна ціна не
   повинна робити товар доступним у публічному ціновому фільтрі.

## Нові необов'язкові параметри

```bsl
ВключатьОписание = ПолучитьБулевоПараметра(
    Параметры, "ВключатьОписание", Ложь);

ОписаниеПоиск = ПолучитьСтрокуПараметра(Параметры, "Описание");

ВключатьОбщееКоличество = ПолучитьБулевоПараметра(
    Параметры,
    "ВключатьОбщееКоличество",
    Не ЗначениеЗаполнено(ПослеКода));
```

На cursor-сторінках без точного загального підрахунку повертати
`total_count = -1`. Next.js трактує від'ємне значення як відсутній підрахунок і
зберігає результат першої сторінки.

## Відбір кандидатів

Для звичайного сортування за кодом запит повинен спочатку створити невелику
тимчасову таблицю:

```bsl
ВЫБРАТЬ ПЕРВЫЕ &ЛимитЗапроса
    Ном.Ссылка КАК Ссылка,
    Ном.Код КАК Код,
    Ном.Наименование КАК Наименование,
    Ном.Описание КАК Описание,
    Ном.ор_НомерПоКаталогу КАК НомерПоКаталогу,
    Ном.ор_Производитель.Наименование КАК ПроизводительНаименование,
    Ном.Родитель.Наименование КАК Подгруппа,
    Ном.Родитель.Родитель.Наименование КАК Группа,
    Ном.Родитель.Родитель.Родитель.Наименование КАК Категория,
    Ном.ФайлКартинки КАК ФайлКартинки
ПОМЕСТИТЬ ВТКандидаты
ИЗ
    Справочник.Номенклатура КАК Ном
ГДЕ
    НЕ Ном.ПометкаУдаления
    И Ном.ЭтоГруппа = ЛОЖЬ
    // Тут додаються активні фасети й текстовий пошук.
    И (&ПослеКода = "" ИЛИ Ном.Код > &ПослеКода)
УПОРЯДОЧИТЬ ПО
    Ном.Код ВОЗР
ИНДЕКСИРОВАТЬ ПО
    Ссылка
```

Для `СортировкаПоЦене`, `ТолькоСЦеной`, `ЦенаОт/ЦенаДо` або
`ТолькоВНаличии` директива `ПЕРВЫЕ` і cursor за кодом на цьому етапі не
використовуються: спочатку формується повний набір кандидатів після дешевих
фасетів.

Окрема таблиця унікальних артикулів не допускає множення рядків:

```bsl
ВЫБРАТЬ РАЗЛИЧНЫЕ
    Кандидаты.НомерПоКаталогу КАК Артикул
ПОМЕСТИТЬ ВТАртикулы
ИЗ
    ВТКандидаты КАК Кандидаты
ГДЕ
    Кандидаты.НомерПоКаталогу <> ""
ИНДЕКСИРОВАТЬ ПО
    Артикул
```

Ціни агрегуються лише для відібраних артикулів:

```bsl
ВЫБРАТЬ
    Ц.Наименование КАК Артикул,
    МАКСИМУМ(Ц.ЦінаПрод) КАК ЦінаПрод,
    МАКСИМУМ(Ц.ЦінаЗакуп) КАК ЦінаЗакуп
ПОМЕСТИТЬ ВТЦены
ИЗ
    Справочник.Ціна КАК Ц
        ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТАртикулы КАК Артикулы
        ПО Ц.Наименование = Артикулы.Артикул
ГДЕ
    НЕ Ц.ПометкаУдаления
СГРУППИРОВАТЬ ПО
    Ц.Наименование
ИНДЕКСИРОВАТЬ ПО
    Артикул
```

Залишки агрегуються лише для кандидатів:

```bsl
ВЫБРАТЬ
    Ост.Номенклатура КАК Номенклатура,
    СУММА(Ост.КоличествоОстаток) КАК Количество
ПОМЕСТИТЬ ВТОстатки
ИЗ
    РегистрНакопления.ТоварыНаСкладах.Остатки(&ДатаОстатков) КАК Ост
        ВНУТРЕННЕЕ СОЕДИНЕНИЕ ВТКандидаты КАК Кандидаты
        ПО Ост.Номенклатура = Кандидаты.Ссылка
СГРУППИРОВАТЬ ПО
    Ост.Номенклатура
ИНДЕКСИРОВАТЬ ПО
    Номенклатура
```

Тимчасові таблиці готуються одним пакетним запитом. `COUNT` (лише коли він
потрібний) і фінальний SELECT повторно використовують той самий
`МенеджерВременныхТаблиц`, тому глобальні агрегати цін та залишків не
повторюються.

## Точні фасети та пошук

```bsl
Если ЗначениеЗаполнено(Категория) Тогда
    ОбщаяЧастьУсловий = ОбщаяЧастьУсловий + Символы.ПС
        + "| И Ном.Родитель.Родитель.Родитель.Наименование = &Категория";
КонецЕсли;

Если ЗначениеЗаполнено(Группа) Тогда
    ОбщаяЧастьУсловий = ОбщаяЧастьУсловий + Символы.ПС
        + "| И Ном.Родитель.Родитель.Наименование = &Группа";
КонецЕсли;

Если ЗначениеЗаполнено(Подгруппа) Тогда
    ОбщаяЧастьУсловий = ОбщаяЧастьУсловий + Символы.ПС
        + "| И Ном.Родитель.Наименование = &Подгруппа";
КонецЕсли;

Если ЗначениеЗаполнено(Производитель) Тогда
    ОбщаяЧастьУсловий = ОбщаяЧастьУсловий + Символы.ПС
        + "| И Ном.ор_Производитель.Наименование = &Производитель";
КонецЕсли;

Если ЗначениеЗаполнено(ОписаниеПоиск) Тогда
    ОбщаяЧастьУсловий = ОбщаяЧастьУсловий + Символы.ПС
        + "| И Ном.Описание ПОДОБНО &ОписаниеПаттерн";
    Запрос.УстановитьПараметр(
        "ОписаниеПаттерн", "%" + ОписаниеПоиск + "%");
КонецЕсли;
```

Для частого пошуку за назвою або описом `%рядок%` потрібен повнотекстовий
індекс 1С чи окремий пошуковий регістр. Звичайний індекс не прискорює leading
wildcard.

## Курсор сортування за ціною

Поточний cursor лише за кодом не можна застосовувати до `Цена, Код`. Новий
непрозорий `next_cursor` повинен містити JSON:

```json
{"price":42.5,"code":"00001234"}
```

Запит наступної сторінки передає його як `КурсорЦены`. Після розбору умова для
ASC:

```bsl
И (
    ЕСТЬNULL(Цены.ЦінаПрод, 0) > &ЦенаКурсора
    ИЛИ (
        ЕСТЬNULL(Цены.ЦінаПрод, 0) = &ЦенаКурсора
        И Кандидаты.Код > &КодКурсора
    )
)
```

Для DESC перший оператор змінюється на `<`, а tie-breaker за кодом залишається
`>`. Порядок завжди має бути стабільним:

```bsl
УПОРЯДОЧИТЬ ПО
    ЦінаПрод ВОЗР, Код ВОЗР
```

або

```bsl
УПОРЯДОЧИТЬ ПО
    ЦінаПрод УБЫВ, Код ВОЗР
```

У відповіді нового режиму додати:

```bsl
Результат.Вставить("cursor_mode", "price_code_v1");
Результат.Вставить("next_cursor", СледующийКурсор);
```

До розгортання підтримки `price_code_v1` у 1С Next.js повинен залишати старий
fallback із вікном 240 товарів. Після появи `cursor_mode` сервер може перейти
на сторінки по 12 товарів без глобального завантаження 240 позицій.

## Фото та чутливі поля

- У списку повертати тільки `ЕстьФото`.
- Не викликати `ПолучитьФотоBase64ПоСсылкеФайла()` всередині циклу каталогу.
- `ЦінаЗакуп` не повертати з публічного response; окремий admin endpoint має
  перевіряти роль на сервері, а не довіряти прапорцю з JSON.
- Самі зображення вже завантажуються окремим пакетним endpoint і кешуються у
  Next.js.

## Перевірка після публікації в 1С

1. Дві послідовні сторінки по 12 товарів не мають пропусків або дублікатів.
2. ASC/DESC із однаковими цінами стабільно продовжуються за `Код`.
3. `total_count` враховує всі активні фільтри, але не cursor.
4. `ТолькоСЦеной` не повертає товари без `ЦінаПрод`.
5. `ВключатьФотоBase64 = Ложь` не читає сховище файлів.
6. Пошук за `Описание` не повертає нефільтровану першу сторінку.
